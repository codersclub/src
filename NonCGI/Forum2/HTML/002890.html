Ладно, всучу тебе немного теории. 2 года назад сам был полный ноль. Перевод буду делать не дословно, а понятно ;-)

An application sends the WM_ERASEBKGND message when the window background must be erased (for example, when a window is resized). The message is sent to prepare an invalidated portion of a window for painting.

Окно получает WM_ERASEBKGND, когда его фон должен быть перерисован (например, при изменении размеров). Оно посылается для того, чтобы окно подготовило недействительную (требующую перерисовки) часть для рисования.

Отступление: в оконном классе (я подразумеваю WNDCLASSEX из Win32, а не класс C++) есть поле для кисти, которая будет применена для отрисовки фона. Если ты не задал её при создании, исправить это можно через SetClassLong. Вообще, вот примерно как можно создать окно с заданной фоновой кистью.

m_wndYourCoolWindow.CreateEx(0,
AfxRegisterWndClass(CS_DBLCLKS, ::LoadCursor(0, IDC_ARROW),
CreateSolidBrush(GetSysColor(COLOR_INFOBK)), 0),
"", WS_POPUP | WS_BORDER, CRect(0, 0, 0, 0), 0, 0);

Объяснять как-то неохота, разберись с Win32 API. Суть в том, что эта кисть применяется для отрисовки фона, и, если ты будешь в OnPaint ещё и рисовать свой другой фон (а так может быть, если ты не подозреваешь, какая кисть класса у тебя установлена), то будет мерцание.

WM_ERASEBKGND позволяет тебе не использовать эту кисть, а отрисовать фон самому или вообще ничего не рисовать (оставить прежний фон), вернув TRUE. Ничего не рисовать при обновлении фона можно также, когда у тебя стоит кисть NULL_BRUSH, как пишет Glite, тогда WM_ERASEBKGND ничего не делает. Если в обработчике WM_ERASEBKGND вернуть FALSE, то windows закрасит фон окна кистю класса.
